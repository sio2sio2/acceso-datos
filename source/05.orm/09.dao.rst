.. _orm-pattern:

Estrategias de diseño
=====================
Cuando analizamos :ref:`conectores <conn>`, ya expusimos que uno de los patrones
de diseño más socorridos es el patrón |DAO|. Al usar un |ORM| podemos seguir
esta misma estrategia sabiendo que hemos sustituido las clases |DAO| por
anotaciones en las propias clases del modelo.

Es conveniente también definir una clase que abstraiga de la conexión, aunque en
el caso de |JPA| es bastante más sencilla que el uso de un pool de conexiones; y
también podemos facilitar la gestión de transacciones aportando un enfoque
funcional como ya hicimos. Por todo ello, sqlutils_ dispone de un módulo
``sqlutils-jpa``:

.. code-block:: xml

   
   <dependency>
       <groupId>com.github.sio2sio2.sqlutils</groupId>
       <artifactId>sqlutils-jpa</artifactId>
       <version>4.2.0</version>
   </dependency>



que implementa una clase gemela a ``JdbcConnection`` llamada
``JpaConnection`` con un uso y una naturaleza semejante, por lo que podemos
ilustrar su uso sin muchos preámbulos: 

.. code-block:: java

   // Propiedades configuradas en tiempo de ejecución.
   Map<String, String> props = new HashMap<>();
   props.put("jakarta.persistence.jdbc.url", "jdbc:sqlite:centro.db");
   props.put("hibernate.show_sql", "true");

   // Configuración en persistence.xml + propiedades dinámicas
   try(JpaConnection jc = JpaConnection.create("MiUnidadP", props)) {
      // Definimos un gestor de transacciones con registros diferidos
      jc.withTransactionManager(Map.of("logging", new LoggingManager()));

      TransactionManager tm = jc.getTransactionManager();

      // La transacción puede generar un error DataAccessException, exactamente
      // el mismo que su correspondiente para conectores.
      tm.transaction(ctxt -> {
         // En este caso obtenemos un EntityManager, no un objeto Connection.
         EntityManager em = ctxt.handle();

         Centro centro = new Centro(11004866L, "IES Castillo de Luna", Centro.Titularidad.PUBLICA);
         em.persist(centro);
         lm.sendMessage(getClass(),
            Level.DEBUG,
            "Centro agregado",
            "Transacción fallida: no puede agregarse el centro"
         );
      });

      // Este método es capaz de devolver lo que devuelve la función del argumento.
      Centro otro = tm.transactionR(ctxt -> ctxt.handle().get(Centro.class, 11004866L));
   }

   // ERROR: Como hemos cerrado el objeto, ya no podemos reutilizarlo
   JpaConnection jc = JpaConnection.get("MiUnidadP");

Obsérvese que el uso de la clase es prácticamente idéntico al de
``JdbcConnection``:

+ Creamos la clase para lo cual necesitamos el nombre de la unidad de
  persistencia y las propiedades de conexión que modifiquen dinámicamente
  los valores estáticos definidos en el |XML|.
+ La clase también implementa un patrón Multiton_ cuya clave identificativa
  es el nombre de la propia unidad de persistencia.
+ Le asociamos un gestor de transacciones que se comporta igual que su
  correspondiente para conectores y, además, soporta los mismos observadores
  (p.ej. ``LoggingManager``).
+ Creamos transacciones con la misma estrategia funcional. La diferencia es que,
  en este caso, el objeto que nos permite operar es un
  :jakarta-persistence:`EntityManager` y no un :java-sql:`Connection <Connection>`.
