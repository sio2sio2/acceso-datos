.. _conn-pool:

*Pool* de conexiones
********************
Abrir una conexión a la base de datos es un proceso costoso en recursos por lo
que, si prevemos que nuestra aplicación abrirá y cerrará varias conexiones, es
conveniente utilizar un :dfn:`pool de conexiones`, que no es más que un
mecanismo que se encarga de administrar un grupo de conexiones a una base de
datos a fin de que puedan ser reutilizadas por diferentes partes de una
aplicación. Esto ahorra al programa el coste de la creación y establecimiento de
conexiones.

.. image:: files/pool.png
   :name: conn-image-pool

En la figura el pool de conexiones tiene abiertas tres conexiones a las bases de
datos, dos de las cuales están siendo usados en la aplicación. Esto significa
que aún hay una libre y que, si ésta necesitara otra, podría usarla sin
necesidad de establecer una nueva conexión con la base de datos.

Para utilizar este mecanismo tenemos dos vías:

+ Usar el mecanismo básico que proporciona, si lo proporciona, el driver |JDBC|
  en cada caso.
+ Lo más recomendable, usar una librería especializada como `HikariCP
  <https://github.com/brettwooldridge/HikariCP>`_.

Creación del gestor
===================
La |API| de |JDBC| se limita a definir una clase |DataSource| con un método
``.getConnection()`` que permite obtener conexiones a partir de ella. No
especifica mucho más, así que no hay ningún mecanismo estándar para crear o
definir los parámetros del *pool*.

Gestor integrado
----------------
Muchos de los :ref:`controladores citados para distintas bases de datos
<conn-drivers>` disponen de un gestor integrado para *pools* de conexiones; y
esta es una estrategia para poder crearlos. Esto, sin embargo, tiene dos
inconvenientes:

- Los gestores difieren entre sí, puesto que a lo único que les obliga la
  especificación es a que proporcionen el objeto |DataSource|: cómo llegar a él
  es libre y, en consecuencia, cada uno utilizará clases y métodos distintos.

  .. attention:: Cuando se pretende usar el gestor integrado :ref:`no debe
     incluirse el scope runtime <scope-runtime>` al cargar la librería, puesto
     que necesitaremos usar directamente métodos que están definidos en dicha
     librería y no en |JDBC|.

- No todos los *drivers* proporcionan un mecanismo para la creación del *pool*,
  y algunos de los que los proporcionan lo hacen de forma parcial (véase más
  abajo el caso de SQLite_).

.. rubric:: SQLite

El driver de SQLite_ es uno de esos casos en los que se proporciona un gestor,
pero bastante inútil. Tengamos también presente que este |SGBD| no está pensado
para la concurrencia, por lo que realiza un bloqueo de la base de datos a nivel
de archivo, esto es, de toda la base de datos; y los pool de conexiones pierden
prácticamente toda su utilidad.

Para obtener un objeto |DataSource| puede hacerse lo siguiente:

.. code-block:: java

   Path dbPath = Path.of(System.getProperty("java.io.tmpdir"), "test.db");
   String dbUrl = String.format("%s%s", dbProtocol, dbPath);

   SQLiteConnectionPoolDataSource ds = new SQLiteConnectionPoolDataSource();
   ds.setUrl(dbUrl);  // No hay que definir usuario ni contraseña.
   
   try(Connection conn = ds.getConnection()) {
      // Utilizamos la conexión.
   }

   try(Connection conn = ds.getConnection()) {
      // Posiblemente se reutilice la conexión anterior,
      // que se marcó como inactiva, al cerrarse.
   }

En el código anterior los dos objetos |DataSource| generan dos objetos
|Connection| distintos. Sin embargo, según la teoría ambos utilizan, en
realidad, la misma conexión a la base de datos, ya que se crean a partir de
un pool de conexiones y al crear el segundo objeto, el primero ya se cerró y,
por tanto, ha dejado disponible la conexión en el pool.

En cambio, si se hubieran generado directamente con |DriverManager|, ambos
objetos conectores estarían asociados a dos conexiones distintas.

.. danger:: Y, sin embargo, lo anterior no es cierto a consecuencia de un `bug
   del controlador de SQLite
   <https://github.com/xerial/sqlite-jdbc/issues/1011>`_, que provoca que
   siempre se abra una nueva conexión, sin reaprovechar las ya establecidas. La
   alternativa, que es:

   .. code-block:: java

      PooledConnection pc = ds.getPooledConnection();
      
      try(Connection conn = pc.getConnection()) {
         // ...
      }

      try(Connection conn = pc.getConnection()) {
         // ...
      }

   tampoco funciona, porque el controlador nunca creará una segunda conexión,
   aunque sea necesaria porque el primer objeto |Connection| sigue activo,
   sino que cierra el primer objeto para aprovechar la conexión con el segundo
   objeto, es decir, que no es capaz de gestionar más que una conexión a la base
   de datos. En el ejemplo no se aprecia el error porque se cierra el primer
   objeto antes de crear el segundo, pero si el código fuera este:

   .. code-block:: java

      try(Connection conn1 = pc.getConnection()) {
         // Aquí podemos usar conn1.

         try(Connection conn2 = pc.getConnection()) {
            // Aquí no podremos usar conn1.
         }

         // Ni aquí tampoco.
      }

   al crearse el objeto ``conn2``, ``conn1`` se cerrará y quedará inútil.

.. rubric:: H2

Nuestro otro |SGBD| de referencia sí soporta concurrencia y sí tiene un gestor
integrado funcional:

.. code-block:: java

   JdbcConnectionPool cp = JdbcConnectionPool.create(url, user, password);
   cp.setMaxConnections(6); // Número máximo de conexions simultáneas abiertas.

   DataSource ds = (DataSource) cp;

   try(Connection conn = ds.getConnection()) {

     // ... Hacemos operaciones con el objeto Connection ...
   }

En este caso, los ejemplos que para SQLite_ no funcionaron correctamente, para
H2_ sí lo harán.

HikariCP
--------
La alternativa, común a cualquier controlador, es usar una librería
especializada como HikariCP_, que tiene `repositorio de Maven
<https://mvnrepository.com/artifact/com.zaxxer/HikariCP>`_.

Su uso es muy sencillo:

.. code-block:: java

   Path dbPath = Path.of(System.getProperty("java.io.tmpdir"), "test.db");
   String dbUrl = String.format("%s%s", dbProtocol, dbPath);
   
   // Configuramos el acceso.
   HikariConfig hconfig = new HikariConfig();
   hconfig.setJdbcUrl(url);
   // En SQLite no hay credenciales de acceso.
   hconfig.setUsername(null);
   hconfig.setPassword(null);
   // Máximo y mínimo de conexiones
   hconfig.setMaximumPoolSize(10);  // Nunca se abrirán más de diez conexiones.
   hconfig.setMinimumIdle(1);       // Al menos habrá una conexión.

   HikariDataSource ds = new HikariDataSource(hconfig);
   HikariPoolMXBean stats = ds.getHikariPoolMXBean(); // Para consultar estadísticas.

   // Como el mínimo es una conexión, ya hay una conexión creada.
   System.out.println(String.format("Conexiones activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 0/1

   try(Connection conn1 = ds.getConnection()) {
      // ...
      System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 1/1
   }

   System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 0/1

   try(Connection conn1 = ds.getConnection()) {
      // ...
      System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 1/1

      try(Connection conn2 = ds.getConnection()) {  // Crea una conexión nueva.
         // ...
         System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 2/2
      }

      System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 1/2
   }

   System.out.println(String.format("activas/totales: %d/%d", stats.getActiveConnections(), stats.getTotalConnections()))  // 0/2

   ds.close();  // Se liberan recursos.

.. warning:: Para el caso particular de usar una base de datos *SQLite* en memoria con HikariCP_,
   la ruta debe ser :file:`file::memory:?cache=shared` y no simplemente
   :file:`:memory:`.

.. _conn-pool-extra:

Gestión avanzada
================
Hasta ahora hemos visto como crear manualmente un *pool* de conexiones, pero
conviene que, al crear un programa, afinemos y abstraigamos al resto de la
aplicación de cómo se crea el *pool* encerrado toda la lógica en una clase como
JdbcConnection_, que propone sqlutils_.

La clase sigue el patrón ya ensayado para :ref:`TransactionManager
<conn-transaction-manager>`: un `patrón Multiton`_ que asocia a cada *pool* de
conexiones una clave\ [#]_:

.. code-block:: java
   :emphasize-lines: 2, 10

   // Pool de conexiones de una base SQLite en memoria.
   try(JdbcConnection jc = JdbcConnection.create("BD", "jdbc:sqlite:file::memory:?cache=shared", null, null)) {
      DataSource ds = jc.getDataSource();
      try(Connection conn = ds.getConnection()) {
         // Hacemos operaciones con la conexión.
      }
   }

   // Aquí ya no podemos usar el pool "DB" porque lo hemos cerrado.
   JdbcConnection.get("DB");  // ERROR: el pool está cerrado.

La última línea muestra cómo puede recuperarse en cualquier momento el *pool*,
siempre que no se haya cerrado.

Ahora bien, ¿cómo obtiene esta clase un objeto |DataSource|\ ? Cuando el
desarrollador no se preocupa de ello, como es el caso, la clase intenta usar
HikariCP_\ [#]_, porque la creación anterior es equivalente a haber añadido un
cuarto argumento al metodo ``create``:

.. code-block:: java
   :emphasize-lines: 5, 8

   // Fábrica de objetos DataSource que usa HikariCP.
   // Opcionalmente se le puede pasar como argumento
   // un objeto HikariConfig para modificar los parámentros predeterminados.
   // Está disponible en el módulo sqlutils-hikaricp.
   DataSourceFactory factory = new HikariCPFactory();

   // Le decimos explícitamente a JdbcConnection cómo
   JdbcConnection.create("BD", "jdbc:sqlite:file::memory:?cache=shared", null, null, factory);

.. warning:: Tenga presente que para que cualquiera de los dos ejemplos
   funcione, necesita importar la dependencia ``sqlutils-hikaricp``.


El objeto puede utilizarse tal como se ha ilustrado para obtener su |DataSource|
y, a partir de él, distintas conexiones |Connection| (que habrá que cerrar
convenientemente), pero es más conveniente aprovechar las posibilidades de
:ref:`TransactionManager <conn-transaction-manager>` y dejar que sea el gestor
de transacción el que se encargue de crear y cerrar la conexión que necesitamos.

.. code-block:: java
   :emphasize-lines: 3, 6

   // Pool de conexiones de una base SQLite en memoria.
   try(JdbcConnection jc = JdbcConnection.create("BD", "jdbc:sqlite:file::memory:?cache=shared")) {
      // Creamos un gestor de transacciones para él y, de paso,
      // registramos un gestor de registros.
      jc.withTransactionManager(Map.of(
         "logging", new LoggingManager()
      ));

      // El gestor de transacciones es sólo accesible a través el gestor de conexiones
      TransactionManager tm = pool.getTransactionManager();

      // La transacción puede generar un error DataAccessException
      tm.transaction(ctxt -> {
         // Conexión asociada a la transacción
         Connection conn = ctxt.handle();

        // Ya podemos realizar operaciones con 'conn'.
      });
   }

.. admonition:: Advertencia

   Las clases JdbcConnection_ y TransactionManager_ son independientes y puede
   usarse una sin necesidad de utilizar la otra, pero están pensadas para usarse
   conjuntamente. Por eso, cuando se ha definido el gestor de transacciones
   correspondiente, el método ``getDataSource()`` genera un mensaje advirtiendo
   de que es conveniente manejar las conexiones a través del gestor, que para
   eso se ha creado.
