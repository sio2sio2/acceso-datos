.. _conn-prog:

Estrategia de programación
**************************
Como se ha podido ver hasta aquí, el acceso de una aplicación a una base de
datos relacional es relativamente sencillo y medianamente semejante sea cual
sea el lenguaje de programación y el |SGBD|. Por tanto, el usar de modo básico
conectores no entraña excesiva dificultad. En realidad, es más complicado,
abstraer al resto del programa del acceso a los datos, de modo que logremos que
en su lógica manipule puramente objetos sin que está manipulación dependa de
cuál es exactamente el soporte de almacenamiento.

Así pues, el propósito a seguir cuando se codifica una aplicación es que todas
las particularidades del acceso a datos estén reducidas a un paquete dentro de
la aplicación (p.ej. llamado *persistence*), fuera del cual no haya otra cosa
que objetos.

A este respecto, podríamos estar tentados de usar el *antipatrón* `God Class`_,
es decir, una única clase que implemente la conexión y todas las operaciones,
pero éste es válido en aplicaciones muy simples que no se prevee que crezcan en
el futuro.

.. seealso:: Todo el código que se describe y comenta en este apartado está en el
   `repositorio de GitHub TestDAO <https://github.com/josem-dam/TestDAO>`_. Se
   recomienda descargarlo para seguir mejor las explicaciones.

Patrón |DAO|
============
Uno de los patrones más usados para lograr la abstracción es el `patrón DAO
<https://www.baeldung.com/java-dao-pattern>`_, que se encarga de tomar los
objetos de la capa de negocio y trasladarlos al soporte de persistencia o
viceversa.  Retomando el ya manido :ref:`ejemplo de centros y alumnos
<ej-centros-alumnos>`:

.. image:: files/DAO.png
   :name: dia-dao

Este patrón básicamente:

+ Define una clase (llamémosla ``Conexion``) que implementa cómo conectarse a la
  base de datos. El objeto correspondiente es el que usa el resto del paquete de
  persistencia para acceder a los datos.
+ Establece una interfaz para definir las operaciones |CRUD|.
+ Define clases (``CentroDAO``, ``EstudianteDAO``) que implementan la interfaz
  anterior para las distintas entidades de la base de datos, que
  forzosamente no tiene por qué ser una base de datos relacional. Un cambio en
  el soporte implica rehacer estas implementaciones, sin alterar el resto de la
  aplicación.
+ Define, dependiendo de la cantidad de entidades, una o varias clases de servicio
  (en el ejemplo, ``AppService``) que utilizan los objetos DAO para acceder a la
  base de datos. Son estos servicios (y no directamente los objetos DAO) los que
  usa el resto de la aplicación para trabajar con los datos.

Por lo general, aunque no forzosamente, cada clase del modelo tendrá asociada
una clase |DAO|. Además, forzaremos a que las clases de nuestro modelo (``Centro`` y
``Estudiante``) deriven de una interfaz que nos asegure que ambas manejan de
igual modo su identificador:

.. admonition:: Aclaración

   Esta interfaz no tiene por qué ser definida exactamente así: podría definirse
   otra que satisfaga también la obligación de implementar las cuatro operaciones
   básicas. Por ejemplo, los métodos que obtienen todas las entidades de un tipo
   podrían devolverlas en forma de |Stream| en vez de |List|.

Como puede verse, tanto las definiciones de las clases como la interfaz
son independientes de cuál sea el soporte de almacenamiento y responden las
primeras a la lógica de la aplicación y la segunda a la necesidad de obtención
de datos almacenados.

Nótese que la interfaz es genérica, porque tiene que servir para cualquier tipo
de objeto que quiera almacenarse en la base de datos. En nuestro ejemplo, se
particularizará para :ref:`Centro <clase-centro>` y para :ref:`Estudiante
<clase-estudiante>`.

Llegamos, por fin, a la parte en la que implementamos la lógica de la
persistencia. Necesitamos, en principio, cuatro clases:

a. Una clase (``Conexion``) que abstraiga de la creación y uso del
   pool de conexiones. Recordemos que ya hemos discutido sobre cómo afinar esto
   :ref:`dentro del epígrafe anterior <conn-pool-extra>`.
#. Dos clases |DAO|: una implemente las operaciones |CRUD| para ``Centro``, y otra
   que haga lo propio para ``Estudiante``.
#. Una clase de servicio (``AppService``) que utilice los objetos |DAO| para
   definir los métodos de acceso a datos que necesita la aplicación.

Si partiéramos de que tenemos un método ``.getConnection()`` en ambas clases
|DAO| que nos proporciona una objeto |Connection| para realizar las operaciones
|CRUD|\ [#]_, la implementación sería trivial, puesto que ya la hemos ensayado
bajo epígrafes anteriores. Por ejemplo:

.. code-block:: java
   :emphasize-lines: 5
   :name: dao-delete

   @Override
   public void delete(Long id) throws DataAccessException {
      String sqlString = "DELETE FROM Centro WHERE id = ?";

      try(Connection conn = getConnection()) {
          try(PreparedStatement pstmt = conn.prepareStatement(sqlString)) {
              pstmt.setLong(1, id);
              boolean deleted = pstmt.executeUpdate() > 0;
              if(deleted) {
                  // Exito
              } else {
                 logger.debug("Borrado fallido: el centro con ID={} no existe", id);
              }
          }
      }
      catch(SQLException e) {
          // Fracaso por algún error de acceso a la base de datos.
          throw new DataAccessException("Borrado fallido del centro con ID=%d: error en la base de datos".formatted(id), e);
      }
   }

.. note:: Obsérvese que hemos actuado como si cerráramos la conexión al acabar
   esta operación. En principio es así, a menos que ``conn`` no sea un objeto
   |Connection| real, sino un objeto *proxy* de |Connection| que tiene
   inutilizado su método ``.close()``. Todo dependerá de qué haga realmente 
   el método ``.getConnection``.

El método (que implementa el método de borrado que declara ``Crud<T>``) no tiene
ninguna dificultad y todos los demás tendrán un aspecto semejante. La miga está
realmente en lo que se ha dejado de hacer:

+ Por un lado, que la operación tenga aparentemente éxito no asegura que se
  lleve a cabo de forma efectiva, ya que esta operación podría formar parte
  de una transacción en la que se realizan varias; y el fracaso de otra podría
  forzar a que la base de datos no confirmara esta operación. Precisamente por
  eso, no hemos hecho que la operación devuelva un valor booleano ni hemos aún
  resuelto qué hacer cuando *deleted* es ``true``.

+ Lo realmente complicado no está resuelto: el método ``getConnection()``. ¿Cómo
  obtenemos la conexión?

Analicemos esto segundo. Ya hemos indicado que la forma de acceder a la base de
datos es mediante un objeto ``Conexion``. Por tanto, en la construcción de los
objetos |DAO| pasaremos este objeto y gracias a él el |DAO| será capaz de
obtener conexiones.  Tenemos dos alternativas:

1. ``Conexion`` ofrece un |DataSource|, por lo que:

   .. code-block:: java

      private Connection getConnection() {
         // cx es el objeto Conexion
         return cx.getDataSource().getConnection();
      }

   En este caso, el |Connection| recibido es una conexión que se usa
   exclusivmente en la operación |CRUD| (*delete*), puesto que tal como está
   escrito el código de la operación, la conexión se abre y se cierra. Esto
   tiene un inconveniente: es imposible que se comparta con otra operación
   |CRUD|, por lo que nunca podremos construir transacciones que involucren
   varias operaciones.  No tener soporte para transacciones es inasumible.

#. ``Conexion`` ofrece un |Connection|, por lo que:

   .. code-block:: java

      private Connection getConnection() {
         return cx.getConnection();
      }

   En este caso, el objeto |Connection| nos lo proporciona ``Conexion`` y
   dependiendo de cómo se haya implementado allí, tendremos o no la limitación
   de no poder hacer transacciones de varias operaciones.
   Por ejemplo, si en ``Conexion`` nos limitáramos a usar un
   |DataSource| para obtener el |Connection| sí existiría la limitación
   anterior, que era inaceptable. Podríamos optar por soluciones más
   sofisticadas que generar un *proxy* que no se cierra\ [#]_, para soslayar la
   limitación de no soportar transacciones, pero inevitablemente tendríamos que
   gestionar cuándo se abren y cierran conexiones, lo cual es un engorro que
   tiene a los fallos.

Como se ve todo es o muy limitado o excesivamente complicado.

.. admonition:: Apunte didáctico

   Antes de continuar, puede implementarse una clase ``Conexion`` muy simple
   que use |DriverManager| o un *pool* manual para ver en concreto estos
   problemas planteados.

Afortunadamente, la solución al problema de gestionar la conexión ya la tenemos:
nuestro :ref:`gestor de transacciones <conn-transaction-manager>`: soporta
múltiples bases de datos (asociando una clave a cada uno de ellos) y múltiples
hilos (por si realizamos programación concurrente). Además, lo programamos de
modo que la conexión que ofrece está protegida contra cierres manuales, ya que
es el gestor el que se encarga de abrirla y cerrarla.  Por tanto, para resolver
el problema y complementar este patrón nos basta con usar en ``Connection`` un
|JdbcConnection| con su |TransactionManager| correspondiente.

Así pues, la clase ``Conexion`` puede ser simplemente una clase *Singleton* que
establezca la conexión gracias a un |JdbcConnection| con |TransactionManager|.
Si se obrara así, la obtención del |Connection| en ``Conexion`` podría ser así:

.. code-block:: java

   /**
    * Devuelve la conexión asociada a la transacción activa.
    * @return La conexión solicitada protegida contra cierres
    * @throws IllegalStateException Si la conexión ya está cerrada o no hay conexión activa.
    */
   public Connection getConnection() {
      // .isOpen() es un método de Conexion que hemos definido también.
      if(!isOpen()) throw new IllegalStateException("La conexión está cerrada");
      // jc es el objeto JdbcConnection que usa Conexion.
      return jc.getTransactionManager().getHandle();
   }

De esto modo, si ya hay una transacción abierta, ``Conexion`` y, en
consecuencia, los objetos DAO, dispondrán de una conexión protegida contra
cierres que no necesitan gestionar: simplemente la usan utilizando exactamente
el código que propusimos más arriba para la operación *delete*. Eso sí, para
cualquier operación necesitaremos abrir explícitamente siempre una transacción.
Por ejemplo, para implementar el método ``.borrarCentro(Centro centro)`` en
``AppService`` tendremos que hacer esto:

.. code-block:: java

   /**
    * Elimina un centro de la base de datos.
    * @param centro El centro que desea ser borrado.
    * @throws DataAccessException Si ocurre un error al realizar la operación.
    */
   public void borrarCentro(Centro centro) {
      // Suponemos que en Conexion hemos definido cómo crear DAOs y transacciones
      // Véase más adelante el código de Conexion
      cx.transaction(ctxt -> {
         var centroDao = cx.getDao(Centro.class);
         centroDao.delete(centro.getId());
      });
   }

Esta es una muy buena (y muy cómoda) solución. Sin embargo, podemos simplificar
todo aún más una vez que hemos decidido usar nuestro gestor de transacciones.
¿Cómo? Simplemente, utilizando en los propios |DAO| también el mecanismo que
ideamos para abrir transacciones con el gestor, en vez de intentar reproducir
fielmente la forma básica de operar con |JDBC|. Siguiendo este camino la
operación *delete* se escribiría así:

.. code-block:: java
   :emphasize-lines: 6
   :name: dao-delete-trans

   @Override
   public void delete(Long id) throws DataAccessException {
      String sqlString = "DELETE FROM Centro WHERE id = ?";

      // cx es el objeto Conexion al que tiene acceso el objeto DAO.
      cx.transaction(ctxt -> {
         Connection conn = ctxt.handle();
         LoggingManager lm = ctxt.getEventListener(LoggingManager.KEY, LoggingManager.class);

         try(PreparedStatement pstmt = conn.prepareStatement(sqlString)) {
             pstmt.setLong(1, id);
             boolean deleted = pstmt.executeUpdate() > 0;
             if(deleted) {
                 lm.sendMessage(
                     getClass(),
                     Level.DEBUG,
                     "Borrado el centro con ID=%s".formatted(id),
                     "Transacción fallida: no llega a borrarse el centro con ID=%s".formatted(id)
                 );
             } else {
                logger.debug("Borrado fallido: el centro con ID={} no existe", id);
             }
         }
      });
   }

La ventaja de escribir la operación de este modo, es que tenemos acceso a todo
el contexto de la transacción, incluidos los observadores que hubiéramos podido
definir. ¿Implica algo abrir la transacción en la implementación de la
operación? Sí, sí implica algo: comodidad. Ya no es necesario declarar
explícitamente la transacción cuando sólo está constituida por una operación.
Además, cuando esté constituida por varias, no habrá problemas, ya que nuestro
gestor de transacción soporta anidación de transacciones.

Finalmente, hay un último detalle. Al cargar un estudiante, la relación exige
que se obtenga también el centro en el que está matriculado. Esta carga podría
ser perezosa o inmediata. En nuestro caso, la carga es inmediata, ya que la
perezosa implicaría la necesidad de implementar un *proxy* que permitiera
diferir la carga hasta el momento en que se pidiera de manera efectiva en
centro.

Además, la carga inmediata la hemos hecho mediante un JOIN y no mediante un
segundo *SELECT*.

   hacerse una idea de cómo puede lograrse.

Conclusiones
============
Toca para cerrar enumerar algunas conclusiones a las que hemos llegado:

+ Es indispensable usar un **pool de conexiones** para mejorar el rendimiento. De
  este modo, podemos cerrar las conexiones según la lógica de la aplicación sin
  preocuparnos  de la penalización del rendimiento.
+ El soporte para **transacciones** es indispensable.
+ Debe separarse el acceso a la fuente de datos del resto de la aplicación. Un
  patrón muy socorrido, sobre todo en bases de datos, es el patrón |DAO|. Sin
  embargo, las clases |DAO| no deben usarse directamente en la parte relativa a
  la interfaz de usuario. Debemos construir una o varias clases de servicio.
+ Gran parte de los retos que debemos resolver al programar con *conectores*, ya
  los resuelven las :ref:`herramientas ORM <orm>`, que trataremos a
  continuación, por lo que en muchas ocasiones en las que o bien el
  programador no es avezado o bien, aunque lo sea, el rendimiento no es crítico,
  es más inteligente utilizarlas.
